## Folder Explanations

- **lib/core/error/** - Centralized error handling with custom exception types for clear error propagation
- **lib/core/network/** - Network abstraction layer for making HTTP requests (currently Google Sheets)
- **lib/core/utils/** - Reusable utility functions (CSV parsing, validation) used across features
- **lib/core/constants/** - Shared constants and configuration values to avoid magic numbers/strings
- **lib/data/model/** - Data classes representing domain entities (FlashCard model)
- **lib/data/datasource/** - Data access layer implementations (remote API, local storage)
- **lib/data/repository/** - Repository pattern implementations that abstract data sources
- **lib/feature/*/domain/usecase/** - Business logic layer containing use cases (one responsibility each)
- **lib/feature/*/presentation/** - UI components and ViewModels following MVVM pattern
- **lib/feature/*/presentation/*_page.dart** - UI widgets (generated by Stitch, logic-free)
- **lib/feature/*/presentation/*_vm.dart** - ViewModels managing state and coordinating use cases
- **lib/feature/*/presentation/*_state.dart** - State classes representing UI states (loading, success, error)
- **lib/feature/*/feature_exports.dart** - Public API for each feature (facilitates clean imports)

## Architecture Flow

```
UI (Page) 
  → ViewModel (VM) 
    → UseCase 
      → Repository 
        → DataSource 
          → Network/Storage
```

## Benefits for AI Code Generation

1. **Predictable Locations** - Each component type has a fixed location, making it easy to find where to add code
2. **Clear Boundaries** - Separation of concerns is explicit through folder structure
3. **Self-Contained Features** - Each feature is isolated, reducing cross-dependencies
4. **Standardized Naming** - Consistent naming patterns (*_page.dart, *_vm.dart, *_usecase.dart)
5. **Single Responsibility** - One file per responsibility makes it easier to understand and modify
6. **Test-Friendly** - Clear separation allows easy unit testing at each layer
